# MS8 Phase 3 Strategy - Pre-Production Mail Safety

Status: Complete (retrospective reconstruction).
Parent milestone: [`docs/milestones/x8_INVOICE_DELIVERY_AUTO_RECEIPTS.md`](../milestones/x8_INVOICE_DELIVERY_AUTO_RECEIPTS.md)
Historical execution date: 2025-11-17.

> This phase boundary was reconstructed on 2026-07-18 from the public-host, aliasing, configuration, and test portions of `dd8092e`. It was not named or checked off as Phase 8.3 during original execution.

## Goal

Make recipient-facing links environment-correct and prevent pre-production delivery tests from contacting real client addresses.

## Reconstruction basis

Primary evidence: `.env.example`, mail configuration, `MailAlias`, application binding, delivery-job integration, `MailAliasTest`, and the alias integration scenario added in `dd8092e`.

## 1. Separate public links from the runtime host

1. [x] Add `APP_PUBLIC_URL` with `APP_URL` as the local default.
2. [x] Generate public invoice links from the explicit recipient-facing base.
3. [x] Keep production configuration responsible for selecting `https://cryptozing.app`.
4. [x] Avoid embedding an accidental localhost or staging host in outbound mail.

## 2. Add configurable catch-all aliasing

1. [x] Add `MAIL_ALIAS_ENABLED` and `MAIL_ALIAS_DOMAIN` configuration.
2. [x] Bind one configured `MailAlias` service.
3. [x] Return the original address when aliasing is disabled or incomplete.
4. [x] Normalize a valid source address into a catch-all local part.
5. [x] Preserve malformed/non-email input rather than inventing a recipient.
6. [x] Keep the behavior deterministic for To and CC recipients.

## 3. Apply containment at the send boundary

1. [x] Resolve recipients from the persisted delivery row.
2. [x] Convert the primary recipient immediately before constructing the outbound send.
3. [x] Convert CC through the same service.
4. [x] Leave the delivery record's original intended recipient intact for audit context.
5. [x] Keep the mailable body and public invoice link unchanged by address rewriting.

## 4. Verify enabled and disabled behavior

1. [x] Prove enabled aliasing rewrites a normal email into the configured catch-all domain.
2. [x] Prove disabled aliasing returns the original address.
3. [x] Prove whitespace/domain normalization.
4. [x] Prove non-email input is returned unchanged.
5. [x] Prove a real delivery job applies aliasing to both To and CC.

## 5. Record the deployment boundary

1. [x] Document the pre-production catch-all intent in mail configuration.
2. [x] Record the configured alias domain in historical PLAN/AGENTS guidance.
3. [x] Make disabling aliasing an explicit production deployment requirement.

## Phase Exit Criteria

- [x] Outbound public links use an explicit environment-appropriate base URL.
- [x] Catch-all aliasing can contain both To and CC delivery in pre-production.
- [x] Disabled aliasing preserves real recipients for eventual production use.
- [x] Unit and feature tests cover the conversion and send-boundary integration.
- [x] Production cutover owns the explicit alias-disable action.

## Historical boundary

Aliasing was a containment mechanism, not delivery idempotency or a production circuit breaker. Later mail hardening added duplicate suppression, retries, send disablement, and truthful provider-state handling.
