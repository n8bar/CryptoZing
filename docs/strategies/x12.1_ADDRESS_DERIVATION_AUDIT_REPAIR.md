# MS12 Phase 1 Strategy - Address Derivation Audit & Repair

Status: Complete (retrospective reconstruction).
Parent milestone: [`docs/milestones/x12_PAYMENT_ADDRESS_ACCURACY.md`](../milestones/x12_PAYMENT_ADDRESS_ACCURACY.md)
Historical execution window: 2025-12-01 through 2025-12-02.

> This phase boundary was reconstructed on 2026-07-18 from the inserted bug-fix milestone, derivation audit, repair command, helper changes, and unit tests. It was not named or checked off as Phase 12.1 during original execution.

## Goal

Prove whether stored invoice addresses belong to the configured watch-only wallet's BIP84 external chain and provide a dry-run-first repair path for mismatches.

## Reconstruction basis

Primary evidence: milestone insertion `fd27880`, bug-fix label `c4c4e85`, audit notes in historical PLAN, and implementation commit `05e22a7` with `wallet:reassign-invoice-addresses`, derivation-helper hardening, and `HdWalletTest` vectors.

## 1. Reproduce and characterize the mismatch

1. [x] Select representative testnet invoices with stored public key, network, address, and derivation index.
2. [x] Derive the expected BIP84 external-chain address independently.
3. [x] Compare the expected address with the stored invoice address.
4. [x] Confirm the affected invoices were derived from the wrong branch shape.
5. [x] Record the issue as a release-blocking payment-address bug rather than routine UX work.

## 2. Lock correct derivation behavior

1. [x] Derive the external branch before applying the invoice index.
2. [x] Normalize supported public extended-key versions for the selected network.
3. [x] Reject a public key whose encoded network conflicts with the configured network.
4. [x] Add known testnet external-chain vectors for indexes 0 and 1.
5. [x] Add known mainnet external-chain vectors for indexes 0 and 1.
6. [x] Add an explicit network-mismatch unit test.

## 3. Add dry-run-first reassignment tooling

1. [x] Add `wallet:reassign-invoice-addresses` with dry-run as the default.
2. [x] Allow targeted inspection by user or invoice ID.
3. [x] Select only invoices with stored addresses and a configured wallet.
4. [x] Re-derive each candidate and report old address, expected address, index, and network.
5. [x] Report checked, matched, updated, skipped, error, cleared-payment, and advanced-wallet counts.
6. [x] Require `--apply` before persisting any address change.

## 4. Protect paid and partially paid invoices

1. [x] Skip invoices with payment activity or paid/partial state by default.
2. [x] Require `--include-paid` to consider those invoices.
3. [x] Require `--reset-payments` before clearing ledger and invoice-level payment metadata.
4. [x] Reset an explicitly selected affected invoice to `sent` when its invalid payment history is cleared.
5. [x] Keep address update and payment reset transactional per invoice.

## 5. Preserve cursor safety during reassignment

1. [x] Reuse an invoice's recorded derivation index by default.
2. [x] Add `--use-next-index` for a fresh sequence from the wallet cursor.
3. [x] Advance an in-memory per-wallet cursor deterministically during the run.
4. [x] Persist the final wallet cursor only during `--apply`.
5. [x] Process invoices in stable ID order.

## Phase Exit Criteria

- [x] The wrong-branch defect is reproducible from stored public data.
- [x] Known vectors lock BIP84 external-chain derivation on testnet and mainnet.
- [x] Network mismatch fails explicitly.
- [x] The repair command defaults to inspection rather than mutation.
- [x] Paid/payment-bearing invoices and wallet cursors require explicit repair options.

## Historical boundary

This phase repaired the current one-key/one-cursor model. It did not preserve historical key lineage or detect key reuse across invoices; MS14 later introduced those safeguards and more auditable correction tooling.
