# MS12 Phase 3 Strategy - Small-Balance Resolution

Status: Complete (retrospective reconstruction).
Parent milestone: [`docs/milestones/x12_PAYMENT_ADDRESS_ACCURACY.md`](../milestones/x12_PAYMENT_ADDRESS_ACCURACY.md)
Historical execution window: 2025-12-04 through 2025-12-05.

> This phase boundary was reconstructed on 2026-07-18 from the small-balance specification and implementation/follow-up commits `5b8a6f9`, `9f6c9ca`, `86e6946`, `f407ec6`, and `684dc56`. It was not named or checked off as Phase 12.3 during original execution.

## Goal

Show the exact USD residual and let an owner explicitly settle an eligible tiny balance through an auditable adjustment without generating contradictory payment requests or alerts.

## Reconstruction basis

Primary evidence: the small-balance notes added in `40fa7aa`, initial implementation in `5b8a6f9`, scaled threshold in `9f6c9ca`/`86e6946`, notification correction in `f407ec6`, and outstanding-sats clamp plus unit test in `684dc56`.

## 1. Stop masking residuals

1. [x] Display exact outstanding USD instead of treating sat tolerance as settlement.
2. [x] Keep automatic paid state dependent only on confirmed USD meeting expected USD.
3. [x] Distinguish display precision from the owner's explicit accounting decision.
4. [x] Do not auto-credit a small residual.

## 2. Define the eligibility threshold

1. [x] Start with an explicit small-balance limit rather than a hidden status tolerance.
2. [x] Scale the final threshold as `max($1.00, min(1% of expected USD, $50.00))`.
3. [x] Reject resolution when no positive balance remains.
4. [x] Reject resolution when the balance exceeds the calculated threshold.
5. [x] Require a usable BTC/USD rate before creating the adjustment.

## 3. Add the owner-initiated resolution action

1. [x] Add an owner-authorized resolve route and controller action.
2. [x] Show the action only when the current residual is eligible.
3. [x] Convert the remaining USD to sats at the latest available rate.
4. [x] Create a confirmed `is_adjustment` payment row for the exact USD residual.
5. [x] Give the row a unique manual-resolution txid and explanatory note.
6. [x] Refresh invoice payment state after insertion.
7. [x] Return explicit success or corrective error feedback.

## 4. Keep notifications truthful

1. [x] Recompute alert thresholds after resolution.
2. [x] Prevent the resolved invoice from sending an underpayment warning.
3. [x] Allow the paid transition to follow the same event/delivery path as other settlement.
4. [x] Add feature coverage that the invoice becomes paid and underpayment nags are skipped.

## 5. Remove residual payment requests after USD settlement

1. [x] Clamp outstanding sats to zero once outstanding USD is zero.
2. [x] Avoid a stale tiny BIP21/QR amount after an adjustment settles USD.
3. [x] Add a focused unit test for the clamped summary.
4. [x] Record the exact-display and explicit-resolution policy in the partial-payments spec.

## Phase Exit Criteria

- [x] Invoice output displays the exact USD residual.
- [x] Small balances never settle automatically.
- [x] Eligible owner resolution leaves a confirmed auditable adjustment row.
- [x] The resolved invoice becomes paid without an underpayment alert.
- [x] Outstanding sats and payment requests are zero after USD settlement.

## Historical boundary

This phase added adjustment creation, not adjustment reversal. Append-only equal-and-opposite reversal was specified and implemented later and belongs to the current partial-payments requirements rather than MS12's original closeout.
