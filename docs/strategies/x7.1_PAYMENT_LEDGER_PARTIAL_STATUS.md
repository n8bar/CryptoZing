# MS7 Phase 1 Strategy - Payment Ledger & Partial Status

Status: Complete (retrospective reconstruction).
Parent milestone: [`docs/milestones/x7_PARTIAL_PAYMENTS_OUTSTANDING_SUMMARIES.md`](../milestones/x7_PARTIAL_PAYMENTS_OUTSTANDING_SUMMARIES.md)
Historical execution date: 2025-11-13.

> This phase boundary was reconstructed on 2026-07-18 around the ledger and state implementation in `b1ec935`. It was not named or checked off as Phase 7.1 during original execution.

## Goal

Represent each detected on-chain transaction independently, preserve its detection-time value, and derive partial/paid invoice state from the accumulated ledger.

## Reconstruction basis

Primary evidence: prospective spec commit `219e489`, its clarification commits through `805b3f9`, implementation commit `b1ec935`, and the contemporaneous watcher feature tests.

## 1. Define the payment-ledger contract

1. [x] Keep the invoice's original amount as the target while adding a one-to-many payment relationship.
2. [x] Represent a payment with:
   1. [x] invoice and transaction identity
   2. [x] optional output index
   3. [x] sats received
   4. [x] detection and confirmation timestamps
   5. [x] confirmed block height
   6. [x] BTC/USD rate and fiat snapshot
   7. [x] optional metadata
3. [x] Prevent duplicate rows for the same invoice and transaction ID.
4. [x] Add `partial` to the invoice lifecycle.
5. [x] Preserve draft and void state from automatic partial-state changes.

## 2. Add persistence and model support

1. [x] Create the `invoice_payments` table with a cascading invoice foreign key.
2. [x] Add the `InvoicePayment` model with explicit fillable fields and casts.
3. [x] Add invoice/payment relationships in both directions.
4. [x] Add aggregate accessors for paid and outstanding sats.
5. [x] Raise the original payment matching tolerance to 100 sats.
6. [x] Add the initial significant-overpayment helpers without mutating ledger rows.

## 3. Record every detected transaction

1. [x] Change detection from one full-payment result to a collection of address transactions.
2. [x] Sum all outputs to the invoice address within each transaction.
3. [x] Retain under-target transactions instead of ignoring them.
4. [x] Upsert each transaction into the ledger by invoice and txid.
5. [x] Persist detection, confirmation, block, sats, and rate metadata.
6. [x] Calculate and store the payment's fiat value from the captured rate when available.
7. [x] Keep repeated watcher runs idempotent for an existing transaction.

## 4. Refresh invoice state from the ledger

1. [x] Sum ledger sats into the invoice-level payment summary.
2. [x] Compare received sats to the original expected target with the 100-sat tolerance.
3. [x] Mark an eligible under-target invoice `partial`.
4. [x] Mark an eligible at-target invoice `paid` and set `paid_at` on the first crossing.
5. [x] Leave draft and void invoices in their existing lifecycle state while still recording detected activity.
6. [x] Save refreshed state after every watcher update.

## 5. Verify and hand off

1. [x] Prove an under-target transaction creates a ledger row and partial state.
2. [x] Preserve confirmation updates for an existing payment row.
3. [x] Preserve the original unconfirmed-as-paid behavior in the contemporaneous full-target test.
4. [x] Leave the ledger aggregates ready for the outstanding-summary presentation phase.

## Phase Exit Criteria

- [x] `invoice_payments` stores distinct detected transactions without duplicates.
- [x] Each row can preserve sats, confirmation metadata, and a USD snapshot.
- [x] Invoice paid/outstanding sats derive from the ledger.
- [x] Eligible invoices can transition to `partial` or `paid` after watcher updates.
- [x] Faked-API watcher coverage proves the first ledger/state behaviors.

## Historical boundary

This phase counted unconfirmed sats toward settlement and used sat-based target comparison. MS12 later made confirmed per-payment USD the settlement authority and introduced `pending` state plus dropped-unconfirmed cleanup.
