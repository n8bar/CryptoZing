# MS8 Phase 1 Strategy - Delivery Workflow & Ledger

Status: Complete (retrospective reconstruction).
Parent milestone: [`docs/milestones/x8_INVOICE_DELIVERY_AUTO_RECEIPTS.md`](../milestones/x8_INVOICE_DELIVERY_AUTO_RECEIPTS.md)
Historical execution window: 2025-11-12 through 2025-11-17.

> This phase boundary was reconstructed on 2026-07-18 from the prospective delivery contract and the persistence/controller/UI portions of `dd8092e`. It was not named or checked off as Phase 8.1 during original execution.

## Goal

Turn an owner request to email an invoice into an authorized, validated, auditable delivery intent.

## Reconstruction basis

Primary evidence: prospective spec `7a087a5`; implementation commit `dd8092e`; and the manual-send scenario in `InvoiceDeliveryTest`.

## 1. Define the delivery record

1. [x] Create `invoice_deliveries` with invoice and user ownership.
2. [x] Record delivery type and lifecycle status.
3. [x] Record recipient, optional CC, and optional owner message.
4. [x] Record dispatch/sent timestamps and error code/message fields.
5. [x] Add invoice and user relationships.
6. [x] Cascade delivery rows when their invoice or user is removed.

## 2. Add the manual-send boundary

1. [x] Add the invoice delivery route and controller.
2. [x] Authorize the owner against the invoice before creating a delivery.
3. [x] Require a client with an email address.
4. [x] Require an enabled public share so the message has a usable recipient link.
5. [x] Validate the optional message and CC-self flag.
6. [x] Resolve recipient and optional owner CC from trusted model data.

## 3. Create and queue the delivery intent

1. [x] Persist a `send` row in `queued` state before dispatch.
2. [x] Record the owner, invoice, recipient, CC, message, and dispatch time.
3. [x] Dispatch one `DeliverInvoiceMail` job for the persisted row.
4. [x] Redirect back to the invoice with visible queued-state feedback.

## 4. Expose delivery history

1. [x] Load invoice deliveries for authenticated invoice show.
2. [x] Display delivery type, recipient, status, and timing.
3. [x] Surface stored failure information for owner review.
4. [x] Keep delivery history private to the authenticated invoice surface.

## 5. Verify the workflow

1. [x] Fake the queue in the owner-send feature test.
2. [x] Assert the queued delivery row contains the expected invoice, message, and CC.
3. [x] Assert the job references the persisted delivery record.
4. [x] Leave actual mail rendering and outcome transitions to Phase 8.2.

## Phase Exit Criteria

- [x] Authorized owners can request invoice delivery from the invoice page.
- [x] Missing client email or public sharing blocks the request safely.
- [x] Every accepted request creates an auditable queued row before job dispatch.
- [x] Delivery history and failures can be reviewed on the invoice.
- [x] Feature coverage proves the queued-intent boundary.

## Historical boundary

This first ledger used `queued`, `sent`, and `failed` states and one row per delivery intent. Later mail milestones added skipped/sending states, business-intent idempotency, retries, circuit breakers, and provider-status reconciliation.
