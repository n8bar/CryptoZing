# MS6 Phase 1 Strategy - Payment-Tracking State

Status: Complete (retrospective reconstruction).
Parent milestone: [`docs/milestones/x6_BLOCKCHAIN_PAYMENT_DETECTION.md`](../milestones/x6_BLOCKCHAIN_PAYMENT_DETECTION.md)
Historical execution date: 2025-11-08.

> This phase boundary was reconstructed on 2026-07-18 around the payment-state foundation in `8d6da28`. It was not named or checked off as Phase 6.1 during original execution.

## Goal

Give invoices enough persistent and visible state to record a detected blockchain payment before introducing the API watcher.

## Reconstruction basis

Primary evidence: commit `8d6da28`, including its migration, invoice model changes, show/print updates, and feature-test additions.

## 1. Add payment-tracking columns

1. [x] Add the received amount in satoshis.
2. [x] Add the current confirmation count.
3. [x] Add the confirmed block height.
4. [x] Add the first payment-detection timestamp.
5. [x] Add the payment-confirmation timestamp.
6. [x] Make each new field reversible in the migration.

## 2. Add model support

1. [x] Add the tracking fields to invoice persistence.
2. [x] Cast detection/confirmation timestamps appropriately.
3. [x] Preserve the original `txid`, `status`, and `paid_at` fields used by the first watcher model.
4. [x] Keep payment state attached directly to the invoice; the separate multi-payment ledger arrived in MS7.

## 3. Surface detected-payment information

1. [x] Add payment amount/confirmation details to authenticated invoice show.
2. [x] Add the corresponding payment details to print output.
3. [x] Keep empty-state rendering usable before any transaction is detected.

## 4. Add regression coverage

1. [x] Extend `InvoicePaymentDisplayTest` for the new payment fields.
2. [x] Confirm payment metadata appears when present.
3. [x] Confirm the invoice surface remains usable when detection fields are empty.
4. [x] Leave API behavior for the next phase's faked-HTTP command tests.

## Phase Exit Criteria

- [x] Invoices can persist sats, confirmation count/height, and detection/confirmation timestamps.
- [x] Show and print can represent the first detected-payment model.
- [x] Empty payment state remains safe and readable.
- [x] The schema is ready for a blockchain detector and watcher command.

## Historical boundary

This phase stored one summarized payment directly on the invoice. MS7 later introduced `invoice_payments` so multiple transactions, partial payments, and per-payment USD snapshots could be represented truthfully.
