# MS19 Phase 2 Strategy — Auth/Password Policy Hardening

Status: Complete. §1–§4 implemented, tested, and eyeball-verified.
Parent milestone doc: [`docs/milestones/19_OB_HARDENING_OPS.md`](../milestones/19_OB_HARDENING_OPS.md)

**Goal:** Implement 419-to-login redirect and site-wide session-expiry logout so expired sessions land users somewhere predictable and recoverable instead of dropping a raw 419 page.

(2FA was originally considered for this phase but has been split into [Phase 8 — Two-Factor Authentication](19.8_TWO_FACTOR_AUTHENTICATION.md). This phase stays narrow.)

## Decisions made

- **419 redirect target:** Return-to-page. Re-auth, then return the user to the page they were on rather than the dashboard — recovery cost is asymmetric (the dashboard is one nav-click from anywhere; the prior page often isn't). The return target is the *page* the form/action lived on (a GET URL), never the POST/PATCH/DELETE route itself — the mutation is never auto-replayed; the user redoes it deliberately with a fresh token. Falls back to the dashboard when no safe same-origin GET target is available.
- **Session expiry duration:** Keep the Laravel default — 120-minute rolling idle timeout. Fits the risk profile (sensitive invoice data, non-custodial wallet); revisit post-launch only if support feedback or threat assessment surfaces a reason to tighten.
- **Logout-on-expiry UX:** Surface an inline message on the login page explaining the session expired. Silent redirects are confusing for users.
- **Scope:** Limited to 419-redirect and session-expiry logout. Other auth surfaces (rate limiting, password policy, 2FA) are out of scope for this phase. 2FA → Phase 8.

## 1. Survey current 419 / session behavior

1. [x] Enumerate every authenticated entry point that can produce a 419 (CSRF token expiry) — forms, AJAX, Livewire/Inertia calls, fetch calls.
2. [x] Group entry points by intent (e.g., login submit, invoice create/edit, wallet settings update, support-access action, etc.).
3. [x] Establish baseline: what does the app currently do today on 419? On session timeout? Note any Laravel default behavior already in place.
4. [x] Lock the redirect-target call. Recorded under Decisions made: **return-to-page**.

**Survey results** (captured here per Exit Criteria):

- **Stack:** Laravel 12 + Breeze (Blade + Alpine; no Livewire/Inertia). 419 sources are `@csrf` form posts and `fetch`/`axios` calls.
- **Two response modalities — the axis that actually drives the work:**
   1. **Full-page form posts** (most authenticated mutations: profile, support-access, invoice/notification/wallet settings, client + invoice CRUD, payment corrections). A server-side redirect to `/login` works.
   2. **AJAX `fetch`/`axios`** (wallet-key validate `resources/js/app.js`, invoice draft + payment-note `resources/views/invoices/show.blade.php`, nav toggle `resources/views/layouts/navigation.blade.php`, rate lookups in invoice create/edit). A server redirect is useless here — the XHR follows it and gets login HTML back. These need a JSON 419 the client catches and turns into a redirect.
- **Intent groups:** guest auth forms (login/register/forgot/reset — a 419 is already unauthenticated, so `/login` is the destination) · logout (already handled) · authenticated mutations (everything else).
- **Baseline:** `bootstrap/app.php` already carries a partial 419 handler — a 419 on `logout` logs out + redirects to `/`; every *other* 419 returns `null` and falls through to Laravel's raw default 419 page. The existing handler already branches on `expectsJson()`, so the JSON-vs-redirect split is an established pattern to extend.
- **Why not Laravel's built-in `intended()`:** it captures the *current request* URL, which for these routes is the POST/PATCH/DELETE endpoint — often no GET counterpart (→ 405), and returning there would imply replaying the mutation. The return target must be the *page* (`url()->previous()`), not the action.

## 2. Implement 419 → login redirect

1. [x] Extend the existing 419 handler in `bootstrap/app.php` (today it only special-cases `logout`) so every 419 is handled instead of falling through to the default page. (`$renderSessionExpired`.)
2. [x] **Full-page requests:** redirect to `/login` (with the expired flag), seeding the post-login return target with the *page* the user was on. Resolution order: `_return_to` (POST body) → `Referer` → dashboard, each validated as a same-origin GET app URL. The body field is what survives when the session/CSRF token is gone (a session-stashed URL would die with the session). Never seed the target with the POST/PATCH/DELETE route — the mutation is not replayed. (`App\Support\ReturnToResolver`.)
   1. [x] Feed `_return_to` via a global submit-listener in `bootstrap.js` that stamps `window.location.href` onto every POST form, rather than a per-form field. Referer is the primary signal (sent in full for same-origin posts — no `Referrer-Policy` is set); the stamped field is the belt-and-suspenders for the Referer-stripped edge. Covers all forms incl. future ones with no per-form wiring. Tradeoff: a no-JS user with a stripped Referer falls back to the dashboard.
3. [x] **AJAX requests** (`expectsJson()` / `X-Requested-With`): return a JSON 419 with a `redirect` hint; a shared client handler (axios response interceptor + a fetch guard) sends the user to the same `/login?expired` surface, passing `window.location` as the return target. No separate AJAX UX — same message, same return-to-page as the form path.
   1. [x] For the two AJAX spots with unsaved input (invoice draft save, payment-note edit), stash the field value in `localStorage` before redirecting and restore it on return. The other four (theme toggle, getting-started, wallet-key validate, rate lookup) carry nothing to preserve — redirect only.
4. [x] After re-auth, route the user to the stashed return target (or dashboard) — `url.intended` wins over first-login persona routing in `AuthenticatedSessionController@store`. Guest auth-form 419s (login/register/forgot/reset) resolve to `/login` naturally — no special handling, no forced login the user didn't need.

## 3. Implement site-wide session-expiry logout

1. [x] Confirm Laravel's session middleware applies the 120-minute rolling idle timeout on every authenticated request. `config/session.php`: driver `database`, `lifetime` 120, `expire_on_close` false — default rolling timeout, no config change.
2. [x] Wire session-expiry detection for both paths. State-changing requests: a 419 redirects to `/login?expired` with the return target per §2.2 (AJAX mirrors §2.3). GET-to-protected-page idle expiry: `redirectGuestsTo` sends the user to `/login?expired` (Laravel already stashes `url.intended`, so return-to-page works) — but only when they carry the `cz_returning` marker cookie, dropped at login and forgotten at logout, which distinguishes an expired session from a never-authenticated first visit.
3. [x] Render the inline expired-session message on `/login` only when the redirect carries the `expired` flag (`request()->boolean('expired')`).

## 4. Tests

1. [x] Feature test: 419 from an authenticated form post redirects to `/login` and stashes the prior page as the return target.
2. [x] Feature test: 419 from an AJAX/JSON request returns a JSON 419 with the `redirect` hint — not an HTML redirect.
3. [x] Feature test: a guest auth-form 419 (login/register/forgot/reset) resolves to `/login` without forcing an unwanted login.
4. [x] Feature test: expired-session redirect surfaces the expired-session message on `/login` (and a plain `/login` does not).
5. [x] Feature test: fresh login from the 419/expired-session redirect lands the user back on the captured page (via `_return_to`); falls back to the dashboard when the captured target is missing, off-origin, or not a GET route.
6. [x] Feature test: GET-idle expiry redirects a returning user (marker cookie) to `/login?expired`; a first-time visitor gets a plain `/login`. Login drops the `cz_returning` marker, logout clears it.
7. [x] Eyeball QA: the two input-preserving AJAX spots (invoice draft, payment-note) restore stashed `localStorage` input after the re-auth round trip. (`localStorage` is outside Laravel feature-test reach — verify in-browser.)

## Exit Criteria

- [x] Full-page 419s redirect to `/login` and return the user to the page they were on after re-auth (dashboard fallback); AJAX 419s return a JSON redirect signal the client acts on.
- [x] Expired sessions log the user out, route to `/login?expired`, surface the expired-session message inline, and return the user to where they were after re-auth — on both the 419 path and GET-idle expiry (the latter gated on the `cz_returning` marker so first-time visitors get a plain `/login`).
- [x] Mutations are never auto-replayed — the return target is always a GET page, never the POST/PATCH/DELETE route.
- [x] Tests cover happy path + edge cases for both flows (form, AJAX, guest auth-form, return-target fallback). `localStorage` restore is eyeball-only (§4.6).
- [x] Survey from §1 captured in this strategy doc (§1 Survey results).
