# MS2 Phase 3 Strategy - Payment & Print UX

Status: Complete (retrospective reconstruction).
Parent milestone: [`docs/milestones/x2_INVOICE_UX_FOUNDATIONS.md`](../milestones/x2_INVOICE_UX_FOUNDATIONS.md)
Historical execution date: 2025-10-29.

> This phase boundary was reconstructed on 2026-07-18 around a distinct payment-output deliverable. It was not named or checked off as Phase 2.3 during original execution.

## Goal

Turn the authenticated invoice record into a payable and printable artifact with BIP21 instructions, copy controls, QR output, and client-facing presentation.

## Reconstruction basis

Primary evidence: commit `73105ec`, with the resulting surfaces included in the combined PR #4 merge `4f40d97`.

## 1. Build payment instructions

1. [x] Add invoice-level Bitcoin URI generation.
2. [x] Include the invoice destination and amount in BIP21 output.
3. [x] Add a copy control for the payment URI/details.
4. [x] Add transaction-ID helper copy to the authenticated invoice surface.

## 2. Build printable output

1. [x] Add the first invoice print route and Blade view.
2. [x] Add server-side QR generation and its package dependency.
3. [x] Render payment instructions and the QR on the print surface.
4. [x] Add the initial thank-you/footer presentation.
5. [x] Organize description and payment content for printed reading order.

## 3. Stabilize invoice identity

1. [x] Add the per-user invoice-number uniqueness migration needed by shared/printed artifacts.
2. [x] Keep the show and print surfaces keyed to the same invoice record and payment details.

## 4. Verify and hand off

1. [x] Confirm authenticated show renders a BIP21 link and copy affordance.
2. [x] Confirm print renders a server-generated QR and payment prompt.
3. [x] Leave print output ready to become the controlled public-share surface in Phase 2.4.

## Phase Exit Criteria

- [x] An authenticated invoice presents actionable BIP21 payment instructions.
- [x] Show and print provide matching address/amount intent.
- [x] Print includes a server-rendered QR and client-facing payment presentation.
- [x] Invoice numbering is stable enough for printable/shared artifacts.

## Historical boundary

This phase used the original invoice payment-address and amount model. MS3 later added direct BIP21/QR regression tests; MS4 tightened amount formatting; MS5 replaced manual destinations with derived addresses.
