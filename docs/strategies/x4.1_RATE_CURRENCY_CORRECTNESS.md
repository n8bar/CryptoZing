# MS4 Phase 1 Strategy - Rate & Currency Correctness

Status: Complete (retrospective reconstruction).
Parent milestone rollup: [`docs/PLAN.md`](../PLAN.md)
Historical execution date: 2025-11-07.
Canonical requirements descendant: [`docs/specs/RATES.md`](../specs/RATES.md)

> This checklist was reconstructed on 2026-07-17; it was not the prospective checklist used during MS4. Checkmarks mean the repository history records the work as shipped. MS4 had no recorded internal phase structure, so `.1` is used only to fit the current strategy-doc naming convention.

Make USD the stable invoice source amount, derive BTC consistently at satoshi precision, and make cached/refresh rate behavior predictable across show, print, and public output.

## Reconstruction basis

The primary evidence is commit `754a955` (PR #21), which introduced the original `docs/RATES.md`, refactored rate/display behavior, and expanded feature tests. The 2025-11-07 changelog entry and historical PLAN commit `a961df0` independently record the milestone as complete.

## 1. Lock the source-of-truth rules

1. [x] Define the entered USD amount as canonical.
2. [x] Derive displayed BTC from USD divided by the applicable rate.
3. [x] Round BTC to no more than eight decimal places.
4. [x] Format USD with two decimal places.
5. [x] Keep BIP21 and QR output aligned with the same displayed BTC amount.

## 2. Normalize rate caching and refresh

1. [x] Normalize cached rate payloads to the shared `rate_usd`, `as_of`, and `source` shape.
2. [x] Treat cached entries as reusable only inside the configured one-hour TTL.
3. [x] Refresh stale or missing cache entries through the live-rate path.
4. [x] Preserve a usable invoice surface when neither cache nor live fetch returns a rate.
5. [x] Keep explicit refresh display-only: recompute BTC/QR values without replacing the stored USD amount.

## 3. Share formatting across invoice surfaces

1. [x] Extract a common invoice-display formatter in the controller.
2. [x] Use it for authenticated show, private print, and public print output.
3. [x] Expose the model BTC formatter for shared use.
4. [x] Keep public output on a fresh-rate-first path with cached fallback.

## 4. Verify MS4

1. [x] Extend rate tests for cache freshness, explicit refresh, and fetch failure.
2. [x] Assert eight-decimal BTC behavior and matching BIP21 output.
3. [x] Verify private and public invoice rendering use the shared display values.
4. [x] Add the rate/currency rules to tracked documentation and link them from PLAN/README.
5. [x] Record Rate & Currency Correctness as complete in PLAN and the changelog.

## Historical boundary

This milestone fixed display and cache correctness for the original single-payment invoice model. It did not define the later per-payment USD snapshots, floating outstanding BTC, or confirmation-aware settlement rules; those were added by MS7 and MS12. The current rates spec is the canonical descendant and may contain requirements that postdate this reconstruction.
