# MS6 Phase 2 Strategy - Mempool Detection & Watcher

Status: Complete (retrospective reconstruction).
Parent milestone: [`docs/milestones/x6_BLOCKCHAIN_PAYMENT_DETECTION.md`](../milestones/x6_BLOCKCHAIN_PAYMENT_DETECTION.md)
Historical execution date: 2025-11-08.

> This phase boundary was reconstructed on 2026-07-18 around the blockchain client, detector, command, and tests in `9efc37f`. It was not named or checked off as Phase 6.2 during original execution.

## Goal

Poll mempool.space for invoice-address activity, identify a full expected payment, record its transaction metadata, and update the invoice through a reusable Artisan command.

## Reconstruction basis

Primary evidence: commit `9efc37f` on the historical `codex/blockchain-watcher` workstream and the contemporaneous `WatchPaymentsCommandTest` scenarios.

## 1. Add blockchain configuration

1. [x] Add `BLOCKCHAIN_CONFIRMATIONS_REQUIRED` with an original default of one.
2. [x] Add configurable mempool.space mainnet and testnet API bases.
3. [x] Add a configurable HTTP timeout.
4. [x] Register the configured `MempoolClient` with the application container.

## 2. Add the mempool.space client

1. [x] Fetch transactions for a network/address pair.
2. [x] Fetch the current chain-tip height.
3. [x] Cache tip height inside the client instance for one watcher run.
4. [x] Return an empty detection input on failed transaction responses.
5. [x] Log network, address/status, and response context on API failures.
6. [x] Reject unknown network API mappings explicitly.

## 3. Detect a matching invoice payment

1. [x] Read the invoice's derived `payment_address`.
2. [x] Calculate the expected payment in satoshis.
3. [x] Walk address transactions and sum outputs to that address within each transaction.
4. [x] Ignore zero-value and under-target transactions.
5. [x] Apply the original five-sat rounding tolerance.
6. [x] For the first matching transaction, return:
   1. [x] txid and sats received
   2. [x] confirmed flag and block height
   3. [x] calculated confirmation count
   4. [x] detection and confirmation timestamps
7. [x] Use chain-tip height to calculate mined confirmations.

## 4. Add `wallet:watch-payments`

1. [x] Scan invoices that:
   1. [x] have a derived payment address
   2. [x] belong to a user with wallet settings
   3. [x] are not void
2. [x] Add `--invoice=` to limit a run to one invoice ID.
3. [x] Process eligible invoices in ID-ordered chunks.
4. [x] Invoke the detector with the wallet network.
5. [x] Apply the result inside a database transaction.
6. [x] Record txid, sats, confirmations, block height, and timestamps.
7. [x] Report processed and updated counts.

## 5. Verify the watcher behavior

1. [x] Fake an unconfirmed matching transaction and prove the command records it.
2. [x] Preserve the original behavior that marked that unconfirmed detection `paid`.
3. [x] Fake a confirmed transaction plus chain tip and prove three confirmations are calculated.
4. [x] Prove confirmation and paid timestamps are recorded for the confirmed case.
5. [x] Keep all tests off the real mempool.space network.

## Phase Exit Criteria

- [x] Public address activity can be queried for configured networks.
- [x] A full expected transaction can be matched to an invoice address.
- [x] Transaction/confirmation metadata persists through the watcher command.
- [x] Faked-API tests cover unconfirmed and confirmed detections.
- [x] No private key or signing behavior enters the application.

## Historical boundary and later correction

Although the configuration exposed a required-confirmations value, the original command's fallback branch marked a matching unconfirmed transaction paid; its test explicitly required that behavior. MS12 later corrected settlement to confirmation-aware state. Partial transactions were ignored here and became first-class records in MS7.
