# MS6 Phase 1 Strategy - Blockchain Payment Detection

Status: Complete (retrospective reconstruction).
Parent milestone rollup: [`docs/PLAN.md`](../PLAN.md)
Historical execution window: 2025-11-08 through 2025-11-11.

> This checklist was reconstructed on 2026-07-17; it was not the prospective checklist used during MS6. Checkmarks mean the repository history records the work as shipped. MS6 had no recorded internal phase structure, so `.1` is used only to fit the current strategy-doc naming convention.

Poll public blockchain data for each derived invoice address, record the detected transaction state, and update invoices automatically without introducing signing keys into CryptoZing.

## Reconstruction basis

The primary evidence is commits `8d6da28`, `9efc37f` (`codex/blockchain-watcher`), `4c75c31`, and `a961df0`, including the original watcher and schedule feature tests. The 2025-11-10 changelog entries record the command integration and scheduling outcome.

The implementation and its tests take precedence where the later historical PLAN summary overstates the original behavior.

## 1. Add invoice payment-tracking state

1. [x] Add invoice fields for received sats, confirmation count, confirmed block height, first detection time, and confirmation time.
2. [x] Expose the initial detected-payment details on invoice show and print output.
3. [x] Keep detection watch-only by querying public address activity; do not add transaction signing or private wallet material.

## 2. Integrate the mempool.space API

1. [x] Add configurable mainnet/testnet API bases, request timeout, and confirmation-count setting.
2. [x] Add a client that fetches address transactions and the current chain tip.
3. [x] Cache the chain tip within one detector run to avoid repeated requests.
4. [x] Log failed API responses and return an empty/no-detection result instead of crashing the watcher.

## 3. Detect invoice payments

1. [x] Sum outputs in each transaction that pay the invoice's derived address.
2. [x] Compare received sats to the original expected BTC amount with a five-sat rounding tolerance.
3. [x] Ignore transactions that do not meet the full expected amount; partial-payment accounting was not part of MS6.
4. [x] Record the matching txid, sats, block height, detection timestamp, confirmation count, and confirmation timestamp when available.
5. [x] Use the chain tip to calculate confirmations for a mined transaction.

## 4. Add and schedule the watcher

1. [x] Add `wallet:watch-payments` with an optional single-invoice filter.
2. [x] Scan non-void invoices that have a derived address and an associated wallet setting.
3. [x] Apply each detection inside a database transaction.
4. [x] Register the command with the application.
5. [x] Schedule it every minute with overlap prevention and background execution.

## 5. Verify MS6

1. [x] Fake the address-transactions API and prove an unconfirmed matching payment records its txid/sats/detection time and marks the invoice paid.
2. [x] Fake a confirmed transaction plus tip height and prove the watcher calculates three confirmations and records confirmation/paid timestamps.
3. [x] Add a schedule test proving the every-minute expression, `withoutOverlapping`, and `runInBackground` flags.
4. [x] Record the watcher command and schedule as complete in PLAN and the changelog.

## Historical boundary and later correction

The original watcher exposed `BLOCKCHAIN_CONFIRMATIONS_REQUIRED` and recorded confirmation metadata, but its state-transition branch still marked an invoice `paid` when the matching transaction was unconfirmed. The original feature test explicitly required that behavior. Therefore MS6 did **not** ship the later confirmation-gated `pending` -> `paid` model, despite shorthand in a historical PLAN summary suggesting that it enforced a confirmation threshold.

MS7 subsequently added partial-payment records and outstanding summaries. MS12 later corrected payment/address accuracy and locked confirmation-aware payment state, including commit `5b8a6f9` (`feat: enforce confirmed USD status and add small balance resolution`). This reconstruction preserves the MS6 implementation boundary rather than attributing those later guarantees to it.
