# MS19 Phase 8 Strategy — Micro-Monetize

Status: Fleshed out — pending review; not yet approved implementation direction.
Parent milestone doc: [`docs/milestones/19_OB_HARDENING_OPS.md`](../milestones/19_OB_HARDENING_OPS.md)
Spec: [`docs/specs/DONATIONS.md`](../specs/DONATIONS.md)

**Goal:** Stand up CryptoZing's first revenue surface — a donation page — as a low-stakes entry into monetization. Makes the "leave room for monetization" posture concrete without committing to fees or paid tiers.

## Scope
**BTC-only.** Donations route to a CryptoZing wallet address; no bank account involved. Fiat donations are backlogged as [`docs/BACKLOG.md`](../BACKLOG.md) item 25.

## Decisions made
- **Surface:** a public, guest-safe `/donate` route in the Laravel app. Ships dark; goes public with the MS21 deploy.
- **Addresses:** derived per donor session from a CZ-owned donation xpub (watch-only), reused until paid. Derive only on explicit donor action — never on page load — with a capped unpaid-address pool as the bot/gap-limit guard.
- **Amounts:** USD presets + custom field, converted at the live rate; chosen amount prefilled in the QR (BIP21).
- **After-pay:** on-page thank-you state when payment is seen. Anonymous — no personal data collected.
- **Architecture:** separate donation model (small donations table + address allocation) reusing the existing derivation and payment-watching services; the invoice domain stays untouched.
- **Operator notification:** one email per recorded donation to a configured operator address (env-configured; address kept out of the repo), triggered when the watcher records the donation — same "payment seen" moment as the thank-you state. Donors receive no mail.
