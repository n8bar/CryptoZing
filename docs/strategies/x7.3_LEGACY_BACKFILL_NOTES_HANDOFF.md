# MS7 Phase 3 Strategy - Legacy Backfill, Notes & Handoff

Status: Complete (retrospective reconstruction).
Parent milestone: [`docs/milestones/x7_PARTIAL_PAYMENTS_OUTSTANDING_SUMMARIES.md`](../milestones/x7_PARTIAL_PAYMENTS_OUTSTANDING_SUMMARIES.md)
Historical execution date: 2025-11-17.

> This phase boundary was reconstructed on 2026-07-18 from the MS7-specific work carried inside `dd8092e`. That commit also delivered MS8 and an MS6 scheduler follow-up; this strategy attributes only the ledger backfill, note, integration, and verification work to Phase 7.3.

## Goal

Move legacy invoice-level payment metadata into the new ledger, add owner annotations, and complete the transition before downstream delivery features consume payment history.

## Reconstruction basis

Primary evidence: `dd8092e`, especially `wallet:backfill-payments`, its feature tests, the payment-note controller/migration, watcher integration, display tests, and the completed MS7 rollup added to historical PLAN.

## 1. Add the legacy payment backfill

1. [x] Add `wallet:backfill-payments`.
2. [x] Select invoices that have legacy txid and received-sats metadata.
3. [x] Process invoices in bounded ID-ordered chunks.
4. [x] Skip invoices that already have ledger rows.
5. [x] Create one ledger row from the legacy:
   1. [x] txid and sats
   2. [x] detection and confirmation timestamps
   3. [x] confirmed block height
   4. [x] invoice rate and calculated fiat amount
6. [x] Report created and skipped counts.

## 2. Make the backfill idempotent

1. [x] Use invoice/txid identity when creating the row.
2. [x] Refuse to create a second row when any payment ledger already exists for the invoice.
3. [x] Prove the missing-row case creates one payment.
4. [x] Prove an existing-row case is skipped.
5. [x] Keep the command safe to rerun after a partial migration.

## 3. Add owner payment notes

1. [x] Add an optional note column to payment rows.
2. [x] Add an owner-authorized update route and controller.
3. [x] Validate a short optional note.
4. [x] Render the note control only on the authenticated invoice surface.
5. [x] Keep public/print history read-only.
6. [x] Add feature coverage for an owner updating a note.

## 4. Finish watcher and display integration

1. [x] Keep watcher updates writing to the ledger rather than only invoice summary columns.
2. [x] Refresh invoice state after each detection pass.
3. [x] Keep show, print, and public summaries fed by the ledger.
4. [x] Preserve payment history for the delivery and receipt work landing in MS8.

## 5. Close the milestone

1. [x] Move Partial Payments & Outstanding Summaries into historical PLAN's completed milestones.
2. [x] Record the ledger, backfill, USD-first summary, and outstanding-targeted payment output.
3. [x] Leave later manual adjustments, alerts, confirmation gating, and residual handling to follow-up work.

## Phase Exit Criteria

- [x] Legacy invoice payment metadata can be represented in `invoice_payments`.
- [x] Repeated backfill runs do not duplicate ledger rows.
- [x] Owners can annotate a payment without changing its financial values.
- [x] Watcher, display, and downstream delivery flows consume the ledger model.
- [x] Historical PLAN records MS7 complete.

## Historical boundary

The backfill carried forward the invoice's stored rate as the payment snapshot because no independent historical detection-time rate existed. Later corrections remained append-only or auditable rather than editing the original on-chain record.
