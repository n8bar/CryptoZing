## Decisions made

- **og:image style: match the favicon mark.** Social cards use the same CZ mark + navy/orange brand palette, not a separate visual language. Replaces the old indigo placeholder card.

- **Favicon: one mark, site-wide.** A single CryptoZing glyph everywhere — no per-surface differentiation, and the per-page `x-emoji-favicon` system is retired. The multi-tab tell-apart benefit was a dev/support nicety, not user-facing.
- **Favicon source: faithful trace of the real logo, not a redraw.** `public/images/CZ.png` is vectorized with `potrace` (color-separated navy `#04254a` / orange `#ff5908` + a white-keyline silhouette) into the master `resources/branding/cz-trace.svg`. Each size carries a **non-linear white keyline** (heavier at small sizes) so the navy C survives dark browser chrome. Pipeline + regen steps in `resources/branding/README.md`.
- **Open-beta refactor placement:** Folded into this phase (§3). Bundled here rather than Phase 5 (Legal Layer) per the same "one coherent visual/brand pass" reasoning.
- **"RC" → "open beta" across active docs.** All active (non-`x`-prefixed) planning, spec, QA, and ops docs use "open beta". Filenames abbreviated "OB": `19_OB_HARDENING_OPS.md`, `21_OB_DEPLOYMENT.md`, `OB_ROLLOUT_CHECKLIST.md`. Scope-label uses become "open beta" plainly; run-up/prep uses get launch framing ("before open-beta launch"). Only code comments and config keep "RC"; completed `x`-prefixed docs keep their historical "RC" (their filename pointers were repointed so links stay valid).

## 3. Keep "RC" / "Release Candidate" out of user-facing copy

"RC" and "Release Candidate" are internal milestone terms; they never appear in user-facing copy. When the RC **deploys** (MS21), the public-facing status is **open beta** — that is the user-facing name for the deployed RC, applied at deploy time. "Pre-release" is a normal lifecycle word and stays wherever it's accurate; it is not RC jargon and is not swapped wholesale.

1. [x] [Agent] Audit user-facing surfaces (`site/` + articles, app + public invoice views, signup/onboarding, mail, error pages, footer) for "RC" / "Release Candidate". Result: **none user-facing.** The only occurrences are internal, unrendered comments — a Blade comment in `resources/views/wallet/settings.blade.php` and a comment in `config/mail.php` — both correct to keep. "pre-release" in `site/{learn,staging}/btcpay-server-alternatives.md` is accurate and stays.
2. [x] [Agent] Regression guard (§5.2): `tests/Feature/UserFacingCopyTest.php` fails if "Release Candidate" appears in user-facing template source (`resources/views` + `site/`), so the jargon can't creep in later.
3. Code comments and config keep "RC" (internal jargon); active worklist/planning docs use "open beta" (filenames abbreviated "OB"); completed `x`-prefixed docs keep their historical "RC".
