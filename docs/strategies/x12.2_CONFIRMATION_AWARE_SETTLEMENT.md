# MS12 Phase 2 Strategy - Confirmation-Aware Settlement

Status: Complete (retrospective reconstruction).
Parent milestone: [`docs/milestones/x12_PAYMENT_ADDRESS_ACCURACY.md`](../milestones/x12_PAYMENT_ADDRESS_ACCURACY.md)
Historical execution date: 2025-12-04.

> This phase boundary was reconstructed on 2026-07-18 from the confirmation strategy, partial-payment spec clarification, implementation commit `5b8a6f9`, and watcher/display tests. It was not named or checked off as Phase 12.2 during original execution.

## Goal

Prevent unconfirmed activity from settling an invoice and derive payment state from confirmed per-payment USD value while remaining safe when an unconfirmed transaction disappears.

## Reconstruction basis

Primary evidence: confirmation strategy introduced in `05e22a7`, the USD-canonical clarification recorded before implementation, and `5b8a6f9` across the invoice model, watcher, dashboard, views, and tests.

## 1. Make per-payment USD the settlement authority

1. [x] Keep the invoice's entered USD total canonical.
2. [x] Preserve each payment's detection-time BTC/USD rate and fiat value.
3. [x] Sum received and confirmed sats for history context.
4. [x] Sum received and confirmed fiat values independently.
5. [x] Calculate outstanding USD from expected USD minus confirmed payment value.
6. [x] Derive a current outstanding BTC request from the outstanding USD and latest available rate.

## 2. Add confirmation-aware state transitions

1. [x] Add `pending` for detected but unconfirmed activity.
2. [x] Keep `partial` for positive confirmed USD below the invoice total.
3. [x] Mark `paid` only when confirmed USD meets or exceeds expected USD.
4. [x] Set `paid_at` only on the confirmed settlement transition.
5. [x] Preserve draft and void lifecycle boundaries.
6. [x] Recompute invoice-level txid, sats, confirmation, and timestamp summaries from surviving rows.

## 3. Clean up dropped unconfirmed transactions

1. [x] Compare stored unconfirmed txids with the address transactions returned by the watcher.
2. [x] Delete an unconfirmed row that is no longer present.
3. [x] Never remove a confirmed ledger row through this cleanup.
4. [x] Recompute received/confirmed totals and invoice state after deletion.
5. [x] Return an invoice from pending to sent when no payment activity survives.
6. [x] Treat a replacement as the live transaction instead of adding its missing predecessor to totals.

## 4. Align display and dashboard behavior

1. [x] Show pending state for unconfirmed payment activity.
2. [x] Keep paid messaging and timestamps gated on confirmed USD.
3. [x] Present received versus confirmed context where useful.
4. [x] Update dashboard open/outstanding calculations for pending/partial state.
5. [x] Keep QR/BIP21 tied to current outstanding USD rather than the creation-time BTC snapshot.

## 5. Verify the corrected state model

1. [x] Prove a full unconfirmed payment leaves the invoice pending and `paid_at` null.
2. [x] Prove confirmed payment value can produce partial state.
3. [x] Prove multiple payments remain distinct and contribute their own snapshots.
4. [x] Prove a missing unconfirmed tx is deleted and state reverts.
5. [x] Keep existing partial-payment, display, delivery, alert, and dashboard coverage aligned.

## Phase Exit Criteria

- [x] Unconfirmed payment activity cannot mark an invoice paid.
- [x] Confirmed per-payment USD value drives partial and paid state.
- [x] `pending`, `partial`, and `paid` communicate distinct confirmation states.
- [x] Dropped unconfirmed rows no longer inflate totals.
- [x] Watcher and display tests lock the corrected behavior.

## Historical boundary

The strategy specified explicit RBF replacement coverage; the contemporaneous feature test directly proved the dropped-unconfirmed case. Per-user confirmation thresholds remained post-RC, and later work refined settlement timestamps and payment-attribution ambiguity.
