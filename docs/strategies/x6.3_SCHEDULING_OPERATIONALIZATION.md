Status: Complete (retrospective reconstruction).

## Goal

Make the watcher discoverable by Artisan and keep it executing every minute without overlapping runs.

## Reconstruction basis

Primary evidence: command-registration commit `4c75c31` and scheduling/test commit `a961df0`, plus the corresponding 2025-11-10 changelog entries.

## 1. Register the command

1. [x] Register application commands through `bootstrap/app.php`.
2. [x] Make `wallet:watch-payments` available to Artisan.
3. [x] Preserve the optional single-invoice invocation for targeted checks.

## 2. Add the Laravel schedule

1. [x] Configure `wallet:watch-payments` to run every minute.
2. [x] Add `withoutOverlapping()` so a slow run cannot stack with the next minute.
3. [x] Add `runInBackground()` so the schedule runner is not blocked by the watcher process.

## 3. Add schedule regression coverage

1. [x] Resolve Laravel's configured schedule in a feature test.
2. [x] Find the event whose command contains `wallet:watch-payments`.
3. [x] Assert the event exists.
4. [x] Assert its cron expression is every minute (`* * * * *`).
5. [x] Assert overlap prevention is enabled.
6. [x] Assert background execution is enabled.

## 4. Close the milestone

1. [x] Update PLAN with the watcher and scheduling outcome.
2. [x] Record command wiring and automated scheduling in the changelog.
3. [x] Leave manual `wallet:watch-payments` available for targeted operational checks.
4. [x] Hand continuous watcher execution to the running Laravel scheduler process.

## Phase Exit Criteria

- [x] Artisan recognizes `wallet:watch-payments`.
- [x] Laravel schedules it every minute.
- [x] Scheduled runs cannot overlap and execute in the background.
- [x] A feature test locks the schedule configuration.

## Historical boundary

This phase registered the Laravel schedule; it did not yet add the dedicated Compose `scheduler` service that continuously runs `php artisan schedule:work`. That container-level operational support landed later and should not be attributed to MS6.
