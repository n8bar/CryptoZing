# MS8 Phase 2 Strategy - Queued Mail & Automatic Receipts

Status: Complete (retrospective reconstruction).
Parent milestone: [`docs/milestones/x8_INVOICE_DELIVERY_AUTO_RECEIPTS.md`](../milestones/x8_INVOICE_DELIVERY_AUTO_RECEIPTS.md)
Historical execution date: 2025-11-17.

> This phase boundary was reconstructed on 2026-07-18 from the queue, mailable, event, listener, preference, and test portions of `dd8092e`. It was not named or checked off as Phase 8.2 during original execution.

## Goal

Render queued invoice email, record the send outcome, and automatically queue the original client receipt when an invoice first crosses into paid state.

## Reconstruction basis

Primary evidence: `dd8092e`, including `DeliverInvoiceMail`, `InvoiceReadyMail`, `InvoicePaidReceiptMail`, `InvoicePaid`, `SendInvoiceReceipt`, the user preference migration, templates, and `InvoiceDeliveryTest`.

## 1. Build the invoice-ready email

1. [x] Add a queueable invoice-ready mailable.
2. [x] Include client and invoice summary context.
3. [x] Include the optional owner message.
4. [x] Link to the invoice's public URL.
5. [x] Add a client-facing Blade template.

## 2. Execute queued deliveries

1. [x] Reload the delivery, invoice, client, and owner inside the job.
2. [x] Select the mailable from the delivery type.
3. [x] Apply To and optional CC recipients.
4. [x] Queue the Laravel mailable through the configured mail transport.
5. [x] Mark the delivery sent and record `sent_at` after successful handoff.
6. [x] Mark the delivery failed and preserve error details when execution throws.

## 3. Emit the paid transition

1. [x] Detect when refreshed payment state crosses from not-paid to paid.
2. [x] Dispatch an `InvoicePaid` event for that transition.
3. [x] Avoid treating every watcher refresh of an already-paid invoice as a new transition.
4. [x] Keep the event usable by later owner/client notification work.

## 4. Queue the original automatic receipt

1. [x] Add the per-user `auto_receipt_emails` preference with its original enabled default.
2. [x] Add the paid-event listener.
3. [x] Reload client and owner state before deciding eligibility.
4. [x] Skip when the client email is absent or the preference is disabled.
5. [x] Skip when a receipt delivery already exists for the invoice.
6. [x] Create a queued `receipt` delivery row.
7. [x] Render the paid-receipt mailable with amount, payment, and public-link context.

## 5. Verify the mail and receipt path

1. [x] Prove an owner request queues the delivery job.
2. [x] Create a payment that crosses the original paid threshold.
3. [x] Prove the paid event/listener creates one receipt delivery.
4. [x] Prove the receipt job is queued from that delivery row.

## Phase Exit Criteria

- [x] Queue jobs can render and send invoice-ready mail.
- [x] Delivery rows record successful or failed execution outcomes.
- [x] A first paid transition emits an event.
- [x] Eligible users receive the original automatic receipt behavior without duplicate receipt rows.
- [x] Feature coverage locks the manual-send and paid-receipt queue boundaries.

## Historical boundary

The original receipt was triggered directly by automatic paid state. Later attribution and truthfulness work required owner review before client receipt sending; the current notifications spec supersedes this policy while retaining the delivery-ledger foundation.
