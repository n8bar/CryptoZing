# MS2 Phase 1 Strategy - Invoice UX Foundations

Status: Complete (retrospective reconstruction).
Parent milestone rollup: [`docs/PLAN.md`](../PLAN.md)
Historical execution window: 2025-10-18 through 2025-11-04.

> This checklist was reconstructed on 2026-07-17; it was not the prospective checklist used during MS2. Checkmarks mean the repository history records the work as shipped. MS2 had no recorded internal phase structure, so `.1` is used only to fit the current strategy-doc naming convention.

Establish the first usable client and invoice lifecycle: CRUD, status changes, USD/BTC presentation, payment instructions, print output, and controlled public sharing.

## Reconstruction basis

The primary evidence is the commit sequence from the project bootstrap (`aa0d045`) through the client/invoice UI merge (`4f40d97`) and rate/show follow-ups (`23205a2`, `9fbb216`, `5208f6b`). The historical PLAN first grouped these already-shipped capabilities into the milestone model on 2025-11-07 and recorded MS2 as complete by commit `a961df0`.

MS2's number does not describe its chronological position relative to MS1: most foundational UX work landed before the later ownership-policy pass was labeled MS1.

## 1. Establish the application and per-user domain

1. [x] Bootstrap Laravel, Breeze, Sail, authentication, and the initial client/invoice schema.
2. [x] Associate clients and invoices with users.
3. [x] Wire client and invoice models, controllers, and authenticated routes.
4. [x] Add client list/create/edit flows plus soft delete, trash, restore, and permanent deletion.

## 2. Build the invoice lifecycle

1. [x] Add invoice create, edit, list, show, trash, restore, and permanent-delete surfaces.
2. [x] Add draft, sent, paid, and void transitions plus reset-to-draft behavior.
3. [x] Add per-user invoice auto-numbering and an explicit invoice date.
4. [x] Add USD amount entry, BTC-rate capture, and live USD-to-BTC calculations on create/edit.
5. [x] Add the invoice show page with rate refresh and a visible "Rate as of" timestamp.

## 3. Add payment and sharing output

1. [x] Generate BIP21 payment URIs and copy controls from the invoice payment details.
2. [x] Add server-rendered QR output and a printable invoice view.
3. [x] Add public-share tokens with enable, disable, rotate, and expiry controls.
4. [x] Serve active public invoices through `/p/{token}` and attach the public noindex header.
5. [x] Recompute display BTC/QR values from the cached or refreshed rate without replacing the stored USD amount.

## 4. Stabilize the first implementation

1. [x] Merge the client UI through PR #3 and the combined client/invoice UI through PR #4.
2. [x] Correct rate refresh so both the displayed amount and timestamp update.
3. [x] Make the show-page BTC amount and QR derive from the shared cached-rate behavior.
4. [x] Record the resulting feature set as Invoice UX Foundations in PLAN.

## 5. Verify MS2

1. [x] Confirm the repository contains the client/invoice migrations, routes, controllers, models, and Blade surfaces named by the historical PLAN.
2. [x] Confirm the public-share implementation contains enable/disable/rotate/expiry behavior and noindex handling.
3. [x] Confirm the merged history preserves the incremental build and corrective rate commits.
4. [x] Defer broad feature regression coverage to MS3, which was created immediately afterward to test these foundations.

## Historical boundary

MS2 established behavior before the project had its later strategy-doc and feature-test discipline. Its original rate rules were tightened in MS4, manual payment-address handling was replaced in MS5, and the public/share, QR, rate, and trash flows received dedicated regression tests in MS3. Those later milestones should not be read back into the original MS2 implementation.
