# MS1 Phase 1 Strategy - Ownership & Access

Status: Complete (retrospective reconstruction).
Parent milestone rollup: [`docs/PLAN.md`](../PLAN.md)
Historical execution window: 2025-11-06.

> This checklist was reconstructed on 2026-07-17; it was not the prospective checklist used during MS1. Checkmarks mean the repository history records the work as shipped. MS1 had no recorded internal phase structure, so `.1` is used only to fit the current strategy-doc naming convention.

Enforce strict per-user boundaries around clients and invoices, including trashed-resource actions, and return a safe, consistent denied-state experience.

## Reconstruction basis

The primary evidence is the implementation and tests in commits `97ce5ba` and `984f664` (PRs #12 and #13), the 2025-11-07 PR #13 entry in [`docs/CHANGELOG.log`](../CHANGELOG.log), and the completed-milestone summary recorded in the historical PLAN by commit `a961df0`.

Where the sources differ in detail, the shipped code and tests at those commits take precedence over later summaries.

## 1. Implement ownership policy boundaries

1. [x] Add `ClientPolicy` and `InvoicePolicy` with owner-only checks for view, update, delete, restore, and permanent deletion.
2. [x] Register the policies with Laravel's authorization layer.
3. [x] Apply resource authorization to the normal client and invoice controller actions.
4. [x] Keep index/trash queries scoped to the authenticated user's records.
5. [x] Explicitly authorize restore and force-delete actions after loading trashed models by ID.
6. [x] Preserve create and list access for authenticated users while preventing cross-user record access.

## 2. Standardize the denied-state experience

1. [x] Route authorization failures through shared exception handling.
2. [x] Add a shared `403` view that does not expose the protected client or invoice.
3. [x] Align the visible denial message with the literal copy expected by the feature test.
4. [x] Follow the initial policy pass in PR #12 with the PR #13 cleanup that simplified controller checks and locked the shared response behavior.

## 3. Verify MS1

1. [x] Add a feature test proving one user cannot view another user's client.
2. [x] Add a feature test proving one user cannot view another user's invoice.
3. [x] Assert both requests return `403` and render the shared friendly denial copy.
4. [x] Confirm the policies themselves cover update, delete, restore, and force-delete even though the original MS1 feature test only exercised cross-user view requests.
5. [x] Record Ownership & Access as complete in PLAN.

## Historical boundary

The original MS1 test added direct regression coverage for cross-user viewing, not an exhaustive authorization matrix. MS3 later added force-delete ownership coverage while hardening the broader feature suite. Historical source code and tests used the term `owner`; the later MS17 issuer-language sweep did not change what MS1 originally implemented.
