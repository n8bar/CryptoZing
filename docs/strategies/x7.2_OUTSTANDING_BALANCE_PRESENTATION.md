# MS7 Phase 2 Strategy - Outstanding Balance Presentation

Status: Complete (retrospective reconstruction).
Parent milestone: [`docs/milestones/x7_PARTIAL_PAYMENTS_OUTSTANDING_SUMMARIES.md`](../milestones/x7_PARTIAL_PAYMENTS_OUTSTANDING_SUMMARIES.md)
Historical execution date: 2025-11-14.

> This phase boundary was reconstructed on 2026-07-18 around the payment-history and outstanding-balance implementation in `fe84d67`. It was not named or checked off as Phase 7.2 during original execution.

## Goal

Make partial settlement understandable on every invoice surface and direct the next payment toward the remaining balance.

## Reconstruction basis

Primary evidence: `fe84d67`, including controller preparation, show/print rendering, and `InvoicePaymentDisplayTest` coverage, plus the 2025-11-14 changelog entry.

## 1. Build one display summary

1. [x] Load invoice payments in stable display order.
2. [x] Calculate expected, received, and outstanding totals.
3. [x] Present USD first while retaining BTC/sats context.
4. [x] Carry the applicable rate and rate timestamp into the summary.
5. [x] Keep the summary usable when payment or rate metadata is absent.

## 2. Target the outstanding amount

1. [x] Add an invoice helper that builds a Bitcoin URI for an explicit amount.
2. [x] Use the calculated remaining BTC amount while a balance remains.
3. [x] Keep the payment address and invoice label aligned with the invoice.
4. [x] Generate QR output from the outstanding-targeted URI.
5. [x] Avoid asking the client to resend the original full amount after a partial payment.

## 3. Update the authenticated invoice view

1. [x] Add an Expected/Received/Outstanding summary block.
2. [x] Display the partial status clearly.
3. [x] Add a payment-history table with transaction identity, amount, timing, and rate/value context.
4. [x] Keep payment controls and outstanding guidance near the summary.
5. [x] Preserve the existing invoice actions and private-state context.

## 4. Update print and public output

1. [x] Add the same expected/received/outstanding model to print output.
2. [x] Add a condensed payment-history table.
3. [x] Keep paid presentation separate from an invoice with a remaining balance.
4. [x] Reuse the public print path without exposing private owner controls.
5. [x] Keep QR and payment instructions consistent with the remaining amount.

## 5. Verify the presentation

1. [x] Prove a partial invoice displays payment history and an outstanding summary.
2. [x] Prove print output includes the payment-history table.
3. [x] Preserve existing BIP21, QR, and payment-metadata assertions.
4. [x] Record the USD-first/outstanding-targeted outcome in the changelog.

## Phase Exit Criteria

- [x] Authenticated, print, and public surfaces can explain a partial payment.
- [x] Payment history is visible with transaction and value context.
- [x] Expected, received, and outstanding totals use one display model.
- [x] BIP21 and QR output target the remaining balance.
- [x] Feature tests lock the partial and print presentation.

## Historical boundary

The original outstanding model was sat-target based and could zero small residuals through tolerance. MS12 later made outstanding USD exact, converted the current USD residual to BTC at view time, and removed display-side dust masking.
