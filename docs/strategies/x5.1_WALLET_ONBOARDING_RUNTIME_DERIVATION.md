# MS5 Phase 1 Strategy - Wallet Onboarding & Runtime Derivation

Status: Complete (retrospective reconstruction).
Parent milestone: [`docs/milestones/x5_WALLET_ONBOARDING_DERIVED_ADDRESSES.md`](../milestones/x5_WALLET_ONBOARDING_DERIVED_ADDRESSES.md)
Historical execution date: 2025-11-07.

> This phase boundary was reconstructed on 2026-07-18 around the runtime wallet/derivation implementation in `10d257e`. It was not named or checked off as Phase 5.1 during original execution.

## Goal

Add watch-only wallet onboarding and replace manually entered invoice destinations with transactionally assigned BIP84-derived receive addresses.

## Reconstruction basis

Primary evidence: commit `10d257e` on the historical `codex/phase-a-wallet` workstream, plus the completed outcome later summarized in PLAN and the 2025-11-08 changelog entry.

## 1. Add watch-only wallet persistence

1. [x] Create one `wallet_settings` record per user.
2. [x] Store only public derivation inputs and cursor state.
   1. [x] Network (`testnet` or `mainnet` in the original implementation).
   2. [x] BIP84 public extended key.
   3. [x] Next derivation index.
   4. [x] Onboarding timestamp.
3. [x] Add the user-to-wallet-setting relationship.
4. [x] Add invoice `payment_address` and `derivation_index` storage.
5. [x] Introduce no private-key or seed-phrase field or normal product path.

## 2. Add wallet settings UX

1. [x] Add `WalletSettingsController` edit/update actions.
2. [x] Add `WalletSettingRequest` validation.
   1. [x] Validate the selected network.
   2. [x] Validate the original supported public-key prefixes.
3. [x] Add `/wallet/settings` routes and navigation.
4. [x] Add the wallet settings Blade form and saved-state feedback.
5. [x] Reset the original cursor to index `0` when the configured public key changes.

## 3. Add the derivation boundary

1. [x] Add Node wallet dependencies.
2. [x] Add the tracked derivation helper later named `derive-address.cjs`.
3. [x] Add `HdWallet` as Laravel's process boundary around that helper.
4. [x] Pass public key, index, and network as explicit derivation inputs.
5. [x] Require a valid address payload and fail on missing script/process/response errors.

## 4. Integrate derivation into invoice creation

1. [x] Redirect users without wallet settings before showing invoice create.
2. [x] Repeat the wallet requirement during invoice submission.
3. [x] Remove manual payment-address validation/input from create and edit.
4. [x] Inside one database transaction:
   1. [x] derive the address at `next_derivation_index`
   2. [x] create the invoice with `payment_address` and `derivation_index`
   3. [x] increment the wallet cursor
5. [x] Render show/print payment output from the derived address.

## 5. Verify and hand off

1. [x] Update existing invoice feature fixtures from `btc_address` to `payment_address`.
2. [x] Confirm invoice creation cannot bypass wallet onboarding.
3. [x] Confirm the tracked implementation contains public extended keys only.
4. [x] Leave pre-existing invoices for the explicit backfill phase.

## Phase Exit Criteria

- [x] Users can save a watch-only public extended key/network.
- [x] New invoices receive a derived address and persisted derivation index.
- [x] Cursor advancement and invoice creation are transactional.
- [x] Manual invoice-address entry is removed from the runtime flow.

## Historical boundary

The original key-change behavior reset one global cursor. Per-key history, cursor resumption, key fingerprints, and stronger validation arrived in later milestones, especially MS14.
