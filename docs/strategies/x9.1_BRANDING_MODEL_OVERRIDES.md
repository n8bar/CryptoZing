# MS9 Phase 1 Strategy - Branding Model & Overrides

Status: Complete (retrospective reconstruction).
Parent milestone: [`docs/milestones/x9_PRINT_PUBLIC_POLISH.md`](../milestones/x9_PRINT_PUBLIC_POLISH.md)
Historical execution date: 2025-11-17.

> This phase boundary was reconstructed on 2026-07-18 from the prospective print/public contract and branding implementation in `318156f`. It was not named or checked off as Phase 9.1 during original execution.

## Goal

Give print and public invoices one resolved billing identity with reusable owner defaults and optional per-invoice overrides.

## Reconstruction basis

Primary evidence: prospective spec `0428384`, implementation commit `318156f`, its migrations/model/controller/form changes, and `InvoicePaymentDisplayTest` additions.

## 1. Define the customizable billing identity

1. [x] Keep billing identity separate from the invoice client's identity.
2. [x] Add owner-level defaults for:
   1. [x] billing name
   2. [x] billing address
   3. [x] billing email
   4. [x] billing phone
   5. [x] invoice footer note
3. [x] Add nullable invoice-level overrides for the same output fields.
4. [x] Preserve room for a customizable heading without adding logo upload.

## 2. Add persistence and model resolution

1. [x] Add billing-default columns to users.
2. [x] Add billing-override columns to invoices.
3. [x] Add fillable/model support for both sets of fields.
4. [x] Resolve effective billing details by preferring a non-empty invoice override over the owner default.
5. [x] Keep unresolved optional fields absent instead of filling misleading placeholders.

## 3. Add owner editing paths

1. [x] Extend profile validation for billing defaults.
2. [x] Add billing-default controls to the original profile settings location.
3. [x] Extend invoice create/edit validation for overrides.
4. [x] Add matching controls to invoice create and edit.
5. [x] Persist only the invoice's own override values on invoice save.

## 4. Render the effective identity

1. [x] Pass resolved billing details to invoice output.
2. [x] Render name, address, email, phone, and footer where present.
3. [x] Keep private print and public invoice output aligned.
4. [x] Preserve invoice number, dates, client, payment, and rate context alongside the new identity.

## 5. Verify default and override behavior

1. [x] Prove print output renders owner defaults.
2. [x] Prove invoice values override their corresponding defaults.
3. [x] Prove show output renders the effective billing details and footer.
4. [x] Keep existing payment-display tests passing with the new fields.

## Phase Exit Criteria

- [x] Owners can persist reusable billing defaults.
- [x] Invoices can persist optional output overrides.
- [x] One model resolution path determines effective billing details.
- [x] Print and public output consume the same effective identity.
- [x] Feature tests prove the override precedence.

## Historical boundary

The controls first lived in Profile and on invoice forms. MS10 moved owner-level invoice/branding defaults to a dedicated Invoice Settings page; Phase 9.3 refined the invoice override authoring surface.
