# MS2 Phase 2 Strategy - Invoice Lifecycle

Status: Complete (retrospective reconstruction).
Parent milestone: [`docs/milestones/x2_INVOICE_UX_FOUNDATIONS.md`](../milestones/x2_INVOICE_UX_FOUNDATIONS.md)
Historical execution window: 2025-10-24 through 2025-10-28.

> This phase boundary was reconstructed on 2026-07-18 from a cohesive commit cluster. It was not named or checked off as Phase 2.2 during original execution.

## Goal

Build the first complete authenticated invoice lifecycle: CRUD/trash recovery, status transitions, per-user numbering, dates, and the initial USD/BTC entry experience.

## Reconstruction basis

Primary evidence: `4b7ad5c`, `cd5624e`, `4f8f616`, `ffaf10f`, and `e814fbe`.

1. [x] Finalize the initial invoice table for the product UI.
2. [x] Add invoice list, create, edit, and trash views.
3. [x] Add invoice validation and persistence through `InvoiceController`.
4. [x] Add invoice soft delete, trash listing, restore, and permanent deletion.
5. [x] Add invoice navigation from the authenticated shell.
6. [x] Add the first BTC-rate service stub needed by invoice forms.

## 2. Add invoice detail and status flow

1. [x] Add the authenticated invoice show page.

## 3. Add numbering and dates

1. [x] Generate the next invoice number per user.
2. [x] Enforce per-user invoice-number uniqueness.

4. [x] Add `invoice_date` through migration, model cast, validation, and create/edit/show UI.

## 4. Add initial USD/BTC form behavior

1. [x] Keep USD amount entry as the primary invoice amount.
2. [x] Add a "Use current rate" endpoint/action.
3. [x] Add live USD-to-BTC calculation on create.
4. [x] Add matching live calculation and refresh behavior on edit.
5. [x] Preserve the original manually entered BTC-address field during this phase; MS5 later replaced it with derived addresses.

## 5. Verify and hand off

1. [x] Confirm invoice CRUD/trash and status routes are wired.
2. [x] Confirm per-user numbering and invoice dates persist through their migrations.
3. [x] Confirm create/edit surfaces show the live conversion behavior.
4. [x] Leave the invoice model/show surface ready for payment instructions and print output.

## Phase Exit Criteria

- [x] Authenticated users can manage the full invoice CRUD/trash lifecycle.
- [x] Invoice statuses, numbering, invoice dates, and due dates are usable.
- [x] Create/edit forms provide the first USD/BTC conversion flow.
- [x] The invoice detail model is ready for payment and print presentation.

## Historical boundary

The form and rate behavior here was foundational rather than final. Phase 2.4 stabilized rate presentation, MS4 locked correctness rules, and MS5 removed manual payment-address entry.
